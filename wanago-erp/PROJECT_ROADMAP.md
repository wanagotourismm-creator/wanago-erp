### Phase 1 — Two-way WhatsApp Inbox ✅ (2026-07-12)
Built `src/modules/whatsapp-inbox/` (conversation list, thread view, reply composer) on **Meta's WhatsApp Cloud API directly** (not Twilio — chosen because replies within a customer's 24h session window are free on Meta's own pricing, whereas Twilio adds a per-message markup on top regardless). Migrated the existing outbound sender (`src/app/api/notify/whatsapp/route.ts`) off Twilio onto the same Meta client (`src/lib/whatsapp/meta-client.ts`) so there's only one WhatsApp integration. New inbound webhook at `src/app/api/whatsapp/webhook/route.ts` (GET verification handshake + POST message/status ingestion, HMAC-signature-verified). New collections `whatsappConversations`/`whatsappMessages`; conversations auto-link to `customers` by phone-number match (full-collection scan — fine at current scale, flagged for an indexed `phoneMatchKey` field if the customer list grows past a few thousand). Admin → Integrations now has a "WhatsApp (Meta Cloud API)" section instead of Twilio.

### Phase 2 — Global Search ✅ (2026-07-12)
Built `src/modules/global-search/` — a `cmdk`-based command palette (⌘K/Ctrl+K, or the header search bar) indexing Customers, Leads, Bookings, Itineraries, Quotations, and Candidates. Client-side approach (confirmed: ~2,000 records total today) — the index is built by calling each module's existing `fetch*` service, so results are automatically scoped to whatever that role already sees on the normal list page; no new Firestore queries, no hosted search service. Index is cached in memory for 3 minutes per session. Selecting a result deep-links into that record's page via a new `?view=<id>` param, wired into all 6 target pages' detail modals.

### Phase 3 — Sales Performance Hub ✅ (2026-07-23)
Unified `incentives`, `sales-team`, `goals` (Sales-department Objectives), `performance/goals`, `performance/reviews` into the existing `/sales-team` page (renamed "Sales Performance Hub" in nav) rather than building a new leaderboard from scratch — `sales-team`'s existing hook already merged leads+bookings+incentives, exactly the prototype this extended. Turned out the "no shared identity type" framing was overstated: all 5 systems already resolved to `Employee.id`. New pure `pickLatestGoalByEmployee`/`pickLatestReviewByEmployee` pick each Sales-department employee's single most-recent HR goal/review; a new `fetchObjectivesByDepartment()` rolls up Sales Objectives into their own panel. Purely computed on render, no new Firestore collection — see CHANGELOG 2026-07-23.

### Phase 4 — Reports buildout (not started)

## Completed workstream: Tools Expansion Release 1 ✅ (2026-07-23)

7 tools from the Wanago ERP Tools Expansion PRD (July 2026), built one at a time in PRD-recommended order: Executive Cockpit → Review & NPS engine → Tally export → Marketing automation/drip → Resource & availability calendar → Traveler companion + SOS → Vendor rate & availability portal. The PRD assumes a "4.0 core" (general ledger, BI/trip-profitability engine, PWA shell) that doesn't exist in this repo — each dependent tool built against current module data instead, with TODOs marking the eventual swap (see `KNOWN_ISSUES.md`). All 7 tools are now complete.

### Tool 1 — Executive Cockpit ✅ (2026-07-20)
Extended the existing `dashboard` module's company-wide (admin/super_admin) view rather than building a new `cockpit` module — it already had revenue/pipeline/forecast/founder-briefing, so a parallel module would have duplicated it (violates the PRD's own "no parallel systems" rule). Added: `computeCashPosition`/`computeGrossMargin`/`computePipelineValue`/`computeArOverdue`/`computeCockpitAlerts` to `dashboard.service.ts` (all pure, unit-tested); `CockpitFilters` (office + date-range) and `AlertsFeed` components; 5-minute auto-refresh in `useDashboard` (paused when tab hidden, skeleton only on first load). Gross margin reads `Booking.profitAmount` (already stamped by Ops on confirm) — no BI engine needed. Two of the PRD's five alert types (resource availability, statutory due-dates) are deferred — no `resources` module or tax calendar exists yet to back them.

### Tool 2 — Review & NPS Engine ✅ (2026-07-20)
New `src/modules/reviews/` (`reviewRequests`/`npsResponses` collections). `Booking.completedAt` + `scheduleReviewRequest()` wired into `updateBookingStatus` (booking.service.ts) — same best-effort/idempotency-checked convention as `createReferralBonusIfEligible`. A new daily cron (`/api/cron/review-requests`, mirrors `daily-reminders`) sends the request once `ReviewSettings.delayDays` has passed, via `sendWhatsAppSmart(purpose: "review_request")` + a new `sendReviewRequestEmail` (notify-server.ts). Public no-login `/review/{token}` page (mirrors `/book/{token}`) + `/api/public/review/{token}` (Admin SDK, same security model as `/api/public/booking-link` and `/api/public/referral`) capture the NPS score; a detractor response auto-creates a `tickets` doc (`sourceType: "nps_detractor"`, `linkedBookingId`, assigned to the original agent) directly via Admin SDK. Reputation dashboard + settings at `/reviews`. **Correction discovered while building this**: `PROJECT_STATUS.md`/`KNOWN_ISSUES.md` previously said WhatsApp had no outside-24h-window template support at all — untrue, `sendWhatsAppSmart`/`template-router.ts` already implements it fully; docs fixed. Real remaining dependency: getting a `review_request` template approved by Meta (business-side, not code).

### Tool 3 — Tally Export / Accounting Bridge ✅ (2026-07-20)
New `src/modules/accounting/tally/` at `/accounting/tally` (added to the existing "Finance" nav group alongside Invoices/Payments/Expenses). Since there's no GL/chart-of-accounts anywhere in the repo, Invoices/Payments/Expenses are treated as implicit Sales/Receipt/Payment vouchers respectively — the pure voucher/XML/CSV-building logic lives in `tally-export.service.ts` (`buildTallyVouchers`/`buildTallyXml`/`buildTallyCsv`, all unit-tested, including the easy-to-get-backwards Tally debit/credit `ISDEEMEDPOSITIVE` convention). New `tallyMappings` (editable ERP→Tally ledger table, mirrors the `whatsapp-templates` admin-table pattern; ships with 5 default system-account rows) and `tallyExports` (a log, not a file store — re-running a period regenerates identically) collections. GST only comes from Invoices (CGST+SGST, back-calculated from the tax-inclusive total — no IGST, since no per-party state-code data exists anywhere to determine inter-state vs intra-state); Payments/Expenses have no tax fields at all to export. Entirely client-side, same `Blob`+`URL.createObjectURL`+`<a download>` mechanism as the existing CSV/PDF export features — no new API route needed. **Process fix discovered while building this**: Tool 2's `/reviews` route was missing its `layout.tsx` (the `<AppShell requiredPage="...">` wrapper every other route uses for sidebar chrome + RBAC gating) — added retroactively; `/accounting/tally` shipped with one from the start.

### Tool 4 — Marketing Automation / Drip Journeys ✅ (2026-07-20)
New `src/modules/journeys/` at `/journeys` (Marketing nav group). `campaigns` was found to have zero send/segment/analytics capability (pure budget/attribution ledger) so journeys is a new module, not an extension — an optional `campaignId` loose link only. Client/server split follows Tool 2's precedent exactly: `journey.service.ts` (client SDK, called from `quotation.service.ts`'s `sendQuotation()` and `booking.service.ts`'s `updateBookingStatus()`) only ever *creates* a `journeyRuns` doc; `journey-engine.server.ts` (Admin SDK, imported only by the new `/api/cron/journey-engine` route) does every actual `sendWhatsAppSmart`/email/notify send, since that needs the Admin SDK and can't run in a browser bundle — so even an "instant" trigger's first action waits for the next daily engine pass, never fires synchronously. Pure `decideAfterStep()` (chain into the next action / defer at a `wait` / complete) is unit-tested in isolation from the Admin-SDK orchestration around it, matching how this codebase generally doesn't unit-test cron routes directly, only the pure logic inside them. New `Customer.marketingOptOut`/`Lead.marketingOptOut` fields (none existed anywhere before — grepped and confirmed), checked at journey-entry and again at every send. Segments approximate "region incl. Gulf NRI"/"spend" with a city-substring filter and the existing `computeCustomerSegment` (newly extracted from `CustomersPage.tsx` into a reusable `computeCustomerSegments()` so journeys don't re-derive the same aggregation) — documented as approximations, not real data. "Create task" reuses `notifyUserServer` (assigns the agent a notification) rather than inventing a 4th collection, since no generic task-tracking system exists anywhere in the app. Added a `"marketing"` `NotificationCategory` (didn't exist before) for these sends.

### Tool 5 — Resources & Availability Calendar ✅ (2026-07-21)
New `src/modules/resources/` at `/resources` (Operations nav group), also reachable inline from `BookingDetailModal`. No vehicle/driver/guide/room-block concept existed anywhere — `Supplier` is a vendor directory (contact + payment terms), not a schedulable entity, so `Resource` carries only an optional loose `supplierId` link, not inheritance. `ResourceAssignment` has its own `startDate`/`endDate` (prefilled from the booking's `travelDate`/`returnDate` when set, always independently required) rather than reading live off the booking, since both of those are independently nullable and can change after a booking is created — this keeps `conflict.service.ts`'s pure `dateRangesOverlap`/`findConflicts`/`computeUtilization` simple (always two real dates to compare). Conflict detection is a **hard block** — the assignment form won't save over an overlap, and names the specific conflicting assignment/blackout. No calendar/timeline library was added (none existed in this repo; only bespoke HR month-grids) — `ResourceCalendarGrid` is a hand-rolled `date-fns` resources×days table that doubles as both "calendar" and "timeline." Closed a gap Tool 1's Executive Cockpit explicitly deferred: `computeResourceAvailabilityAlerts` (in `dashboard.service.ts`) now flags a `(type, office)` group where every active resource has a conflict in the coming week. Per-itinerary-day assignment and PWA/Traveler-Companion surfacing are out of scope (no date field on `ItineraryDay`, no PWA shell, Traveler Companion is Tool 6 — not built).

**Bug fix discovered while re-running rules-tests for this tool** (unrelated to Resources itself): `firestore.rules`'s manager-lookup helpers (`isOwnerOrManagerOfEmployee`/`isOwnerOrEitherManagerOfEmployee`/`isEitherManagerOfEmployee`) dot-accessed `emp.reportingManagerId`/`emp.functionalManagerId` directly instead of `.get(key, null)` — for the common case of an employee with neither field set, Firestore rules throw an evaluation error (not a falsy value), denying the whole request rather than just failing that one OR-branch. This silently broke attendance/leave/regularization/asset-request writes for any employee without a manager assigned, in production, since before this session. Fixed to match the `.get(key, default)` convention used everywhere else in this file for optional fields.

### Tool 6 — Traveler Companion + SOS ✅ (2026-07-21)
New `/portal/customer/companion` page inside the **existing** customer portal (`src/app/portal/customer/`, `src/modules/portal/`) rather than a third auth mechanism — the portal already has a real phone+referral-code customer identity via `requirePortalAuth`, which is exactly what a legitimate SOS needs (an anonymous `/book/{token}`-style link wouldn't be). New `src/modules/companion/` (`tripCompanions`/`sosEvents` collections). Itinerary join is genuine, not approximated: `src/lib/package-itinerary-sync.ts` already keeps `Package.itineraryId`/`Itinerary.packageId` in sync both ways, so `booking.packageId → package.itineraryId → itinerary` is reliable — a booking with no package simply shows no itinerary. `Resource` gained a `phone` field (none existed) so the guide/driver/vehicle contact card can offer a `tel:` link. `selectRelevantBooking()` (the one pure, unit-tested piece — 8 cases incl. active-beats-upcoming, open-ended return date, past-trip exclusion) picks which of a customer's bookings the page shows: an in-progress trip (today within travelDate..returnDate) always wins over a later upcoming one. All reads/writes go through new Admin-SDK routes (`GET /api/portal/customer/companion`, `POST .../opt-in`, `POST .../sos`) — mirrors `booking-requests/route.ts` exactly, no client Firestore path for the customer side at all. SOS creates a `sosEvents` doc and best-effort notifies the booking's assigned agent + every admin/operations user via the existing `notifyUserServer`, with a Google Maps link. Staff-side visibility is a small read-only `BookingSosHistory` component (same integration idiom as Tool 5's `BookingResourcesSection`) inside `BookingDetailModal`, with a client-side "Mark Resolved" action gated to Admin by a new `firestore.rules` block (`sosEvents`: read open to authenticated staff, write Admin-only; `tripCompanions` has no dedicated rule at all — Admin-SDK-only, nothing ever queries it client-side). Post-trip Review/NPS hook needed no new wiring — Tool 2's `scheduleReviewRequest` already fires on booking completion, independent of this page. Out of scope, documented not faked: offline/PWA itinerary caching (no manifest/service-worker infra exists), background/continuous live-location tracking (only updates on toggle or explicit refresh — no background-geolocation infra exists), a per-destination emergency-hotline database (only the generic India 112 number + the business's own phone are real data), and non-admin SOS resolution (Admin-only for this pass).

### Tool 7 — Vendor Rate & Availability Portal ✅ (2026-07-23, final tool)
New `src/modules/vendor-portal/` (`vendorRates`/`vendorAvailability` collections), plus a new `Supplier.vendorPortalToken` field. `Supplier` (`src/modules/suppliers/`) had zero login/credential capability and no rate/pricing field of any kind — confirmed by direct code read before planning. Two portal-auth families already existed in this repo: the identified/logged-in customer/partner portal (hardcodes a `PortalType` union, requires a `referralCode`+phone login) and the anonymous long-lived token-link (`/book/{token}`, `/review/{token}` — the token itself is the entire security boundary, no login at all). Since inventing vendor credentials was out of scope, this tool reuses the **second** pattern exactly: `generateVendorPortalToken()` (same idempotent `crypto.randomUUID().replace(/-/g,"").slice(0,24)` shape as leads' `generateBookingLink`), a "Generate Vendor Link"/copy/WhatsApp-share block in `SupplierDetailModal` (replicated from `LeadDetailModal`'s existing booking-link UI verbatim), and a new `GET`/`POST /api/public/vendor/[token]/route.ts` structured exactly like `booking-link/[token]/route.ts`. The vendor-facing `/vendor/{token}` page is deliberately plain (modeled on `/review/{token}`'s simple fetch-on-mount shell, not `/book/{token}`'s heavily animated marketing page) since this is an internal B2B utility, not a conversion touchpoint. Vendors can only **add** rates/availability, never edit/delete their own past submissions — keeps the public route free of per-row auth logic; staff correct/remove anything from the new `/vendor-rates` page (Operations nav, `operations`/`admin`/`super_admin` RBAC), which also lets staff enter rates directly for suppliers who never touch the portal, with a "Vendor Submitted" vs "Staff Entered" badge (`submittedByVendor`) on every row. Reused Tool 5's `dateRangesOverlap` (from `conflict.service.ts`) for a **non-blocking** `findOverlappingAvailability` hint — unlike `Resource`, a vendor's declared capacity isn't a single schedulable instance being double-booked, so there's no case for `findConflicts`-style hard-blocking here. Also built `findApplicableRate` (most-recent-rate-active-on-a-date lookup), scoped only to make a future Quotation/Package costing integration cheap to wire in later — not wired in yet. `firestore.rules` mirrors `suppliers`' existing dedicated block (read: any authenticated user; write: `operations`/`admin`/`super_admin` only) rather than the plain catch-all Resources relies on, since rate/pricing data is commercially sensitive the same way Supplier's billing fields are; vendor writes bypass this entirely via the Admin SDK. Out of scope, documented not faked: real vendor login/credentials, auto-wiring rates into Quotation/Package costing, multi-currency (`currency: "INR"` hardcoded), and an approval/rejection workflow for vendor submissions (v1 is a plain visible list, no `pending/approved/rejected` status).
