# WANAGO ERP — PROJECT STATUS

_Last updated: 2026-07-23_

## Current Phase

Core ERP build-out is substantially complete (~40 feature modules). **Tools Expansion Release 1** (PRD, July 2026) — 7 tools beyond the 4.0 core, built one at a time — is now complete as of Tool 7; see `PROJECT_ROADMAP.md`. The **non-AI platform gaps** workstream (WhatsApp inbox, global search, sales performance unification, reports buildout) ran to completion first. A separate, independent workstream is handling AI/Gemini integration (`aiassistant`, `hr-chat`, `ai-assistant`, `ai-core`) — not tracked in this doc.

**Note:** the Tools Expansion PRD assumes a "4.0 core" (general ledger, BI/trip-profitability engine, PWA shell) that does not exist in this repo. Per-tool decision: build against current module data with TODOs marking the eventual GL/BI swap, rather than blocking Release 1 on building those first — see `KNOWN_ISSUES.md`.

## What's built

- **CRM/Sales**: Leads, Customers, Suppliers, Bookings (Finance+Ops dual-approval), Quotations (Finance approval, branded PDF, auto-email, convert-to-booking), Itineraries, Itinerary Brochures (branded PDF), Packages, Campaigns
- **Finance**: Invoices, Payments, Expenses
- **HR/People**: HRMS (attendance w/ geofencing + selfie check-in, payroll, leaves, org chart, document hub, regularization), Recruitment/ATS, Onboarding, Onboarding Training (interactive walkthroughs + TTS voiceover), Performance (goals + reviews), Goals (company OKRs), Leave Policy, ESS
- **Sales performance**: Incentives (computed tiered incentive/bonus engine), Sales Team dashboard (leads+bookings+incentives merged), personal role-scoped dashboards (Sales, Marketing, HR, Finance, Operations)
- **Executive Cockpit** (Tools Expansion Release 1, Tool 1): extends the existing admin/super_admin company-wide dashboard (`dashboard` module) rather than a separate module — cash position, gross margin %, open pipeline value, and AR-overdue tiles; an alerts feed (overdue invoices, negative-margin bookings) that deep-links to the source record; office + date-range filters; 5-minute auto-refresh. Reads live off `bookings`/`payments`/`expenses`/`leads`/`invoices` (no GL/BI engine to read from yet — see note above)
- **Review & NPS Engine** (Tools Expansion Release 1, Tool 2): `src/modules/reviews/` — a booking completing auto-schedules a post-trip review request (configurable delay, default 2 days) sent via WhatsApp (`sendWhatsAppSmart`, purpose `review_request`) + email; public no-login `/review/{token}` page captures an NPS 0–10 score + comment; promoters get a Google review link, detractors auto-create a "Service Recovery" ticket (`tickets.sourceType: "nps_detractor"`) assigned to the original agent. Reputation dashboard at `/reviews` (NPS trend, response rate, promoter/passive/detractor split by destination/agent) + a Settings tab for delay/thresholds/Google review link. Real dependency: a `review_request`-purpose WhatsApp template needs Meta approval + registration in Admin → WhatsApp Templates before WhatsApp sends work outside the 24h window — email always works regardless.
- **Tally Export / Accounting Bridge** (Tools Expansion Release 1, Tool 3): `src/modules/accounting/tally/` at `/accounting/tally` — one-click export of approved invoices, all payments, and paid expenses for a chosen period as Tally-compatible XML (masters for party/expense-category ledgers + Sales/Receipt/Payment vouchers, correct debit/credit convention) or a plain reconciliation CSV. GST (CGST+SGST only, back-calculated from Invoice's tax-inclusive total) flows only from Invoices — Payments/Expenses have no tax field to export. An editable `tallyMappings` table maps expense categories + the 5 system accounts (Sales/CGST/SGST/Cash/Bank — assumed to already exist in Tally) to real ledger names; unmapped categories still export (flagged, not blocked). `tallyExports` logs every run (counts + unmapped list), no file content stored. 100% client-side — same `Blob`+`URL.createObjectURL` download pattern as the existing CSV/PDF exports, no new server route.
- **Marketing Automation / Drip Journeys** (Tools Expansion Release 1, Tool 4): `src/modules/journeys/` at `/journeys` — config-driven journeys (trigger → ordered steps) for quote-sent, quote-unaccepted-after-N-days, and trip-completed. Steps: wait(days), send WhatsApp (`sendWhatsAppSmart`), send email, notify the assigned agent (the "create task" action — reuses `notifyUserServer`, no generic task system exists), add-to-segment (informational only — segment membership is always computed live from its filters, never stored). New `Customer.marketingOptOut`/`Lead.marketingOptOut` fields, enforced both at journey-entry and at every send. New daily cron `/api/cron/journey-engine` — **steps advance at most once/day, never instantly** (no sub-daily cron exists in this app, and `sendWhatsAppSmart` needs the Admin SDK so it can only ever run server-side, never from the browser code that detects the trigger). Segments filter on real fields only (destination/source/computed customer segment/city-substring) — "region incl. Gulf NRI" and "spend" from the PRD aren't backed by real data, so segments approximate with a city-substring proxy and the existing booking-derived customer segment, documented as such. Analytics (entered/sent/replied/converted/revenue) are denormalized counters on each Journey, rolled forward by the cron; reply/conversion are proxies (a WhatsApp reply / a booking created after entering), and email "opened" isn't tracked at all (no pixel infra).
- **Resources & Availability Calendar** (Tools Expansion Release 1, Tool 5): `src/modules/resources/` at `/resources` — a registry of vehicles/drivers/guides/room-blocks (capacity + office, optional loose link to a Supplier vendor record), assigned to bookings with **hard conflict detection** (`findConflicts`, blocks the save and names the specific conflicting assignment/blackout — not just a dismissible warning), a resources×days calendar/timeline grid, blackout periods (maintenance/leave), and a utilization report. Assignment also reachable straight from `BookingDetailModal`, not just the standalone `/resources` page. Closes a gap Tool 1 explicitly deferred: the Executive Cockpit now has a "low resource availability" alert (every active resource of a type/office busy in the coming week). `ResourceAssignment` carries its own `startDate`/`endDate` independent of the booking's (which can be null/change later) — prefilled from the booking, always required. No calendar library was added — a hand-rolled `date-fns` grid, matching this project's existing hand-rolled HR calendars.
- **Traveler Companion + SOS** (Tools Expansion Release 1, Tool 6): new `/portal/customer/companion` page inside the existing customer portal (real phone+referral-code identity, not an anonymous token link) — day-by-day itinerary (joined `booking.packageId → package.itineraryId → itinerary`, only for bookings that have a package), guide/driver/vehicle contact cards with `tel:` links (`Resource.phone`, new field), a generic-112 + business-phone emergency contacts list, an opt-in live-location toggle, and a prominent **SOS button** (confirm-gated, captures + reverse-geocodes location client-side via the existing `src/lib/geo.ts`, then POSTs to `/api/portal/customer/sos`). SOS creates a `sosEvents` doc and best-effort notifies the booking's assigned agent plus every admin/operations user with a Google Maps link — staff see a read-only SOS history + "Mark Resolved" action inside `BookingDetailModal`. `selectRelevantBooking()` picks which booking to show (an in-progress trip always wins over a later upcoming one). All portal reads/writes are Admin-SDK-only (`requirePortalAuth`), same pattern as the rest of the customer portal.
- **Vendor Rate & Availability Portal** (Tools Expansion Release 1, Tool 7 — final tool): new `src/modules/vendor-portal/` (`vendorRates`/`vendorAvailability` collections) plus a new `Supplier.vendorPortalToken` field. Since `Supplier` has no login/credential field of any kind, the vendor-facing side reuses the anonymous long-lived token-link pattern already established by `/book/{token}`/`/review/{token}` (not a new `PortalType`) — a "Generate Vendor Link" button in `SupplierDetailModal` mints a token (same idempotent `crypto.randomUUID()` shape as leads' `generateBookingLink`), and the vendor visits `/vendor/{token}` (no login) to submit rates (service/unit/rate/validity window) and bulk availability (resource label/date range/units), add-only — they can't edit or delete past submissions, only staff can, from the new `/vendor-rates` staff page (Operations nav, admin/operations RBAC). Staff can also enter rates/availability directly for a supplier that never uses the portal. `findOverlappingAvailability` (reused `dateRangesOverlap` from Tool 5's `conflict.service.ts`) surfaces a non-blocking "Overlaps" hint — unlike Tool 5's `Resource`, a vendor's declared capacity isn't a single schedulable instance, so there's nothing to hard-block. `findApplicableRate` picks the most-recent rate active on a given date. A `VendorRatePicker` modal (browse/filter real submitted rates, click one to insert) is now wired into both `QuotationForm.tsx` ("Add from Vendor Rate" next to the AI "Suggest Items" button — the AI drafter's own line items default to ₹0 "for you to fill in from real rates," which this now does) and `PackageForm.tsx` ("Look up Vendor Rate" next to Cost Price, fills a starting-point number staff can adjust). This is a manual browse-and-pick tool, not an automatic computation — no field anywhere reliably joins a line item/Package to a specific `VendorRate` (both `serviceName` and `lineItems[].description` are free text), so staff judgment still decides what applies.
- **Ops/Collab**: Team Space (Slack-style chat), Tickets, Assets, Call Logs, Notifications, Digests, Help Center, Training (formal programs)
- **Admin**: Users/RBAC/permissions, Integrations (secret management for Twilio/Resend/Gmail/Google TTS/Anthropic), System Health, Usage & Quotas (Firebase + Supabase), Activity log, Trash/restore, Org Explorer, Data Export, Offices, Holidays, Announcements

## Known gaps (see `PROJECT_ROADMAP.md` for the plan)

- Global Search doesn't cover Suppliers/Packages/Invoices (out of scope for Phase 2; revisit if needed) and is client-side only — would need a hosted index (Algolia/Typesense) if data volume grows an order of magnitude past ~2,000 records
- Sales performance data lives in 3 disconnected-ish places (`incentives`, `sales-team`, `performance/*`) with real but partial cross-referencing; no unified hub view
- `reports` module is a single 291-line page, not a proper module (no schemas/services/hooks split); only one computed report (customer retention) beyond raw per-module dumps
- No general ledger or BI/trip-profitability engine (the Tools Expansion PRD's "4.0 core"); Executive Cockpit, Tally export, and the Vendor portal all read module data directly with TODOs left for the eventual swap
- Two of the Executive Cockpit's five PRD-listed alert types (low resource availability, statutory due-dates) aren't modeled — no `resources` module or tax-calendar exists yet to back them
- The Review & NPS engine's WhatsApp send needs a `review_request`-purpose template approved by Meta and registered in Admin → WhatsApp Templates — a business-side action, not a code gap; email works regardless of whether that's done
- Tally Export only supports CGST+SGST (no IGST — no per-customer/office state-code data exists to determine inter-state vs intra-state); Payments/Expenses carry no GST/TDS data to export since the source modules don't have those fields; the Sales/CGST/SGST/Cash/Bank ledgers must already exist in Tally with correct configuration — only party and expense-category ledgers are auto-created
- Marketing Automation journeys only advance once/day (no sub-daily cron exists on this Vercel tier, and sends can only run server-side); no birthday/anniversary trigger (no such field exists on Customer); no email open tracking (no pixel infra); "region incl. Gulf NRI" and "spend" segment filters are approximations (city-substring, computed booking-based segment), not real region/LTV data
- Resources doesn't do per-itinerary-day assignment (only per-booking — `ItineraryDay` has no date field, and Itinerary isn't linked to Booking at all, so per-day granularity would be materially bigger scope)
- **Fixed a real pre-existing bug found while re-running rules-tests for Tool 5**: `firestore.rules`'s `isOwnerOrManagerOfEmployee`/`isOwnerOrEitherManagerOfEmployee`/`isEitherManagerOfEmployee` dot-accessed `emp.reportingManagerId`/`emp.functionalManagerId` directly — for any employee record missing one of those (most of them; both are optional), Firestore rules throw an evaluation error instead of a falsy value, denying the *entire* request rather than just that branch. Fixed to use `.get(key, null)` like every other optional-field check in this file. This affected real attendance/leave/regularization/asset-request writes in production, not just this session's changes.
- Traveler Companion has no offline/PWA support (no manifest/service-worker infra exists in this repo — out of scope for this pass, documented not faked); no background/continuous live-location tracking (only updates on toggle or explicit refresh — no background-geolocation infra exists); emergency contacts are limited to the generic India 112 number + the business's own phone, not a fabricated per-destination hotline directory; itinerary days show numbered labels only (no per-day date field on `ItineraryDay`); only an Admin can resolve an SOS event (not yet widened to a broader ops role)
- Vendor portal has no real vendor login/credential system (token-link is the deliberate substitute); vendor rates are wired into Quotation/Package costing only as a manual picker (`VendorRatePicker`), not an automatic computation — no field reliably joins a line item/Package to a specific rate, so nothing computes a total from vendor rates automatically; no multi-currency (`currency: "INR"` hardcoded); no approval/rejection workflow for vendor submissions — v1 is a visible list staff can edit/delete directly, no `pending/approved/rejected` status; vendors can only add rates/availability, never edit or delete their own past submissions; `Booking.profitAmount`/`OpsApprovalModal`'s suggestion still isn't backed by vendor rates (no reliable resource↔rate join exists)
