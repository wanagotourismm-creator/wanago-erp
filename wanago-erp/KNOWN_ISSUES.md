# KNOWN ISSUES

_Last updated: 2026-07-23_

## Gaps (not bugs, but real functional holes)

- ~~WhatsApp has no message templates yet~~ **Corrected 2026-07-20 — this was wrong.** `src/lib/whatsapp/template-router.ts`'s `sendWhatsAppSmart()` already routes outside-24h-window sends to an approved Meta template (HSM) registered per-`purpose` in the `whatsappTemplates` collection (Admin → WhatsApp Templates), falling back to free text inside the window. Discovered while building the Review & NPS engine (Tools Expansion Release 1, Tool 2), which uses it directly. The one real remaining dependency is business-side: getting each needed template (e.g. `review_request`) approved by Meta and registered in the admin panel — not a code gap.
- **No global search.** Nothing indexes across customers/bookings/leads/itineraries/quotations/candidates today; each module is siloed to its own list/filter UI.
- ~~Sales performance data is fragmented... no shared schema or identity type~~ **Corrected 2026-07-23.** The 5 systems (`incentives`, `sales-team`, `goals.Objective.ownerId`, `performance/goals.employeeId`, `performance/reviews.employeeId`) already shared one identity space all along (`Employee.id`) — confirmed via direct code read, not the blocker this entry previously claimed. Unified into the Sales Performance Hub (`/sales-team`) — see CHANGELOG 2026-07-23. Remaining gap: `reviewerId`/`postedById` (Firebase Auth uid, "who wrote this") is a genuinely different identity space from `Employee.id` ("who this is about"), bridged only optionally via `Employee.userId` — irrelevant to the hub itself but unresolved if that space needs unifying elsewhere later.
- ~~`reports` module is thin~~ **Resolved 2026-07-23.** Restructured into `types/services/hooks/components/pages`. Two computed reports now exist with `recharts` visualizations: Customer Retention (Supabase mirror, now with a bar chart) and the new Firestore-native Sales Trend report (moving average, MoM/YoY). The 6 raw-table tabs (Employees/Attendance/Leaves/Payroll/Recruitment/Performance) still just export a filtered dump of their source module's data — no aggregation was added there, only the module structure changed.
- **No internal general ledger — deliberately.** The Tools Expansion PRD assumed a "4.0 core" including a general ledger; confirmed with the user this should NOT be built internally, since `src/modules/accounting/tally/` already exists to feed the real external ledger (Tally) — a second one would duplicate it. The Executive Cockpit still reads cash/margin/pipeline numbers straight off `bookings`/`payments`/`expenses`/`leads`/`invoices` (see `dashboard/types/index.ts`'s TODO). Two of the PRD's five cockpit alert types (low resource availability, statutory due-dates) — resource availability is now modeled (Tool 5); statutory due-dates still aren't (no tax-calendar concept exists).
- **BI/trip-profitability is now real, but narrow.** As of 2026-07-23, `src/modules/profitability/`'s `computeTripProfitability()` gives Booking a real computed profit (`totalAmount − Package.costPrice − linked Expenses`) instead of a fully-manual number, and `Expense` can now be linked to a `bookingId`. It does NOT include vendor rates or Resources (no reliable join from `ResourceAssignment`/`Resource` to `VendorRate` exists — both free text, no shared key) and does NOT feed the Executive Cockpit's `computeGrossMargin` (kept to real Ops-confirmed `profitAmount` only, to avoid mixing confirmed and estimated figures in a headline metric).
- **Vendor rates are wired into Quotation/Package costing only as a manual picker, not a computation.** As of 2026-07-23, `VendorRatePicker` lets staff browse/insert real vendor-submitted rates in `QuotationForm.tsx`/`PackageForm.tsx`, but nothing auto-fills or auto-totals from them — no field anywhere reliably joins a Quotation line item or Package to a specific `VendorRate` (both free text), so a real computed integration remains a materially bigger effort. `Booking.profitAmount`/`OpsApprovalModal`'s suggestion still isn't backed by vendor rates at all (no reliable resource↔rate join exists there either).
- **No approval/rejection workflow for vendor-submitted rates/availability.** A vendor's submission via `/vendor/{token}` shows up immediately as a plain list item at `/vendor-rates` (tagged "Vendor Submitted"); staff can edit/delete it there, but there's no `pending`/`approved`/`rejected` status or review gate before it's considered live data.
